## 1. What WhyBuddy Is

WhyBuddy is an open-source **AI product rehearsal engine**. Its core value proposition:

> Enter one sentence. In ~5 minutes, receive a full product rehearsal — requirements doc, design doc, task breakdown, system architecture Mermaid diagram, UI mockup prompts, traceability matrix, and a prompt pack — all with an evidence trail proving the system didn't fake it.

It occupies a niche no existing tool fills: one-sentence-in → complete SPEC package out, with multi-route planning, multi-role agent collaboration, real-time 3D observability, Docker-sandboxed execution, and human takeover governance. It competes against Dify/n8n/CrewAI/LangGraph but beats all of them on the "idea → deliverable spec" flow.

The system is a web app (React + Vite + Three.js frontend, Express + Socket.IO server) with a separate Docker-based execution sidecar called **Lobster Executor**. It also ships a standalone **Skill package** (`skills/whybuddy.zip`) that can be dropped into any agent host (Trae, Claude Code, etc.) for the same spec-generation flow without the full web app.

Scale: 5,457 project files, 575,591 TypeScript lines, 921 test files, 303 `.kiro/specs` directories.

## 2. Full Feature Catalog

### 2.1 Core Product Rehearsal (the main flow)

| Feature | Description |
|---|---|
| **One-sentence input** | User enters a free-form product idea, optionally attaching a GitHub repo URL, local files (PDF/Word/Excel/image), or screenshots. |
| **GitHub deep ingestion** | If a repo URL is included, the system reads files, symbols, and interface contracts from the real repo, falling back gracefully when private/inaccessible. |
| **Clarification dialogue** | LLM-driven clarification loop identifies blocking vs. non-blocking missing info, asks targeted questions, produces a `clarified_brief.json` with goal, constraints, and numbered success criteria. |
| **Multi-route planning** | Generates three route variants (standard / deep / upgraded), compares risk level and cost level for each, presents a confirmation gate before proceeding. Equivalent to "show me plan A, B, and C". |
| **SPEC tree generation** | Core output: a structured tree of modules (requirements → design → tasks → evidence) validated by schema, stable ID normalization, and an invariant guard (unique root, parent-reachable, no cycles, requirements cover success criteria, each node has evidence). LLM-generated with deterministic fallback if LLM fails. |
| **Spec documents** | Auto-generates `requirements.md`, `design.md`, `tasks.md` from the SPEC tree. Each doc uses EARS-format acceptance criteria (Easy Approach to Requirements Syntax). |
| **Traceability matrix** | Bidirectional mapping: requirement ↔ design ↔ task ↔ evidence ↔ test case. Exported as `traceability_matrix.json`. |
| **Architecture diagram** | Mermaid diagram rendered deterministically from the SPEC tree — not generated by an image model. |
| **UI mockup generation** | Per-module UI preview images generated by a configurable image model (gpt-image-2, Gemini, etc.), labeled "preview · unverified". Anti-fake-image checks via `check_previews_real.py`. |
| **Prompt pack** | A curated set of LLM prompts for each SPEC module, ready to use for implementation. |
| **Handoff/export** | ZIP + Markdown export of the full spec package including interface contracts (draft), open items log, checks ledger, visual previews with provenance labels, and handoff manifest. |

### 2.2 Multi-Agent Collaboration (FSD Fleet)

| Feature | Description |
|---|---|
| **Decision Gate** | Before expensive generation, classifies work as "simple" (single agent path) vs. "complex" (multi-role brainstorm). Falls back to single agent on timeout or failure. |
| **Brainstorm board** | Multi-role collaborative mode: discussion, voting, division of labor, and audit mode among specialized roles. |
| **Role fleet** | 10 on-demand roles: Single Agent, Planning, Architecture, Execution, Audit, UI, Critic, Grounding, Synthesizer, Decision. Roles share a common tool proxy. |
| **Critic role** | Triggered by ambiguity, real-repo risk, or evidence gaps. Finds holes, weak evidence, and overconfident assumptions. Trace written to `companion_log.json`. |
| **Grounding role** | Triggered when claims need real code citations. Reads the repo and forces actual source references into the output. |
| **Synthesizer role** | After multi-role work, merges proposals, confidence scores, and dissenting opinions into a single route decision. |
| **Dynamic organization** | Instead of a fixed 18-agent roster, generates a custom org structure (departments, roles, skills, MCP config, concurrency strategy) per task. Falls back to a safe default if generation fails. |

### 2.3 Runtime Observability

| Feature | Description |
|---|---|
| **3D office scene** | React Three Fiber (Three.js) 3D office where each AI agent is a visible character. Agents move, animate, and react as work progresses. The scene reflects real runtime state. |
| **Task cockpit** | The `/tasks` page is a full-screen cockpit with Overview / Execution / Artifacts tabs, session-isolated real-time state, and streaming card display of in-flight work. |
| **Streaming stage progress** | Every stage emits events to the server event bus → Socket.IO → real-time store → frontend. Derived node status drives "pending / generating / done / failed / retry" display. |
| **Replay** | Full replay of any past session: events are persisted to the job store and can be replayed with session-level isolation. |
| **Audit timeline** | `AuditTimeline.tsx` and `AuditPanel.tsx` surface the full audit chain: what ran, exit codes, LLM calls, role decisions, and evidence citations. |
| **Cost dashboard** | Real-time token usage and estimated cost per session, per agent, per model. Budget alerts (WARNING → CAUTION → CRITICAL → EXCEEDED) with configurable response strategies (LOG / REDUCE_CONCURRENCY / DOWNGRADE_MODEL / PAUSE_TASK). |

### 2.4 Human-In-The-Loop Governance

| Feature | Description |
|---|---|
| **Takeover points** | Explicit pause-and-ask gates at: clarification, route confirmation, permissions/budget, risk review, and delivery review. The system never silently fails or auto-proceeds past these. |
| **Wait / resume** | Missions can wait for human approval (`WAITING_INPUT` state) and resume via API (`resume()`). |
| **Autopilot L1-L5 levels** | Execution promise model: L1 = route suggestion; L2 = partial auto-execution with human gates; L3 = standard task auto-close loop; L4 = limited-domain high automation; L5 = open-domain (research concept only). A mission can downgrade its level mid-run. |
| **Approval workflow** | Missions that require external approvals (budget, permission, external write) route through an explicit approval state. |
| **Risk actions** | Structured runtime risk observations — ambiguity, missing info, tool failure, permission, cost, side-effects, quality — each with a defined response action. |

### 2.5 Execution Layer

| Feature | Description |
|---|---|
| **Lobster Executor** | Separate sidecar service (port 3031) that runs tasks in Docker containers. Three modes: `mock` (deterministic), `native` (host process), `real` (Docker). HMAC-signed callbacks, credential scrubbing, seccomp security policy, resource limits (memory 512m, CPU 1.0, PIDs 256). |
| **Docker sandbox** | Each execution job runs in an isolated Docker container (`node:20-slim` or a custom AI image). Supports pause/resume of containers. |
| **Skill jobs** | Executor has a `skill-registry` that maps Skill IDs to container images + entry points. Skills are hot-pluggable. |
| **Log batching** | Executor batches stdout/stderr log lines and streams them back to the server via a retry buffer with property-tested delivery guarantees. |
| **Screenshot capture** | Executor can capture screenshots from running containers (headless browser tasks). |

### 2.6 Quality Assurance

| Feature | Description |
|---|---|
| **Checks ledger** | Every quality gate writes the script run, exit code, and output to `checks_ledger.json`. Nothing is hand-waved. |
| **EARS content check** | Validates that requirements use EARS-format acceptance sentences ("The system shall…"). |
| **Invariant guard** | SPEC tree must pass structural invariants: unique root, all nodes parent-reachable, no cycles, max depth, requirement-to-success-criteria coverage, evidence on every node. |
| **Preview audit** | `check_previews_real.py` catches fake images (model returned an error but wrote a placeholder), duplicate images (copy-pasted to fill module count), and fallback images. |
| **Companion trace** | `companion_log.json` records what the Critic and Grounding roles flagged — must be non-empty to pass the companion check gate. |
| **Provenance labels** | Every artifact has a source label: `llm`, `llm_fallback`, `template`. UI mockups are labeled "preview · unverified". Interface contracts are labeled "draft · unverified". |

### 3.2 Process Model

1. **dev:all** starts three processes concurrently: `vite --host` (frontend), `tsx watch server/index.ts` (server), lobster-executor.
2. **Production**: `docker compose up` boots `whybuddy-app` (server + bundled frontend on 3000:3001) + `whybuddy-mysql`. Lobster executor is opt-in separately (Docker-in-Docker complexity).
3. **Browser-only mode** (`dev:frontend`): Frontend works standalone on GitHub Pages. No server, no `.env`. Uses deterministic mock/template fallback for all LLM-backed features.

## 4. Key Cross-Cutting Concepts / Domain Vocabulary

Understanding this vocabulary is essential for reading WhyBuddy code.

| Term | Definition |
|---|---|
| **Mission** | A top-level work unit initiated by a user. Contains a `Destination`, chosen `Route`, runtime state (`Drive State`), agent fleet assignment, and result artifacts. Maps to a row in the DB. |
| **Destination** | What the user wants to achieve: goal, constraints, success criteria, missing info, expected deliverables. The structured output of the clarification phase. |
| **Route** | A specific execution path for a mission. Each mission has multiple candidate routes (standard/deep/upgraded) compared by risk level and cost level before one is selected at a confirmation gate. A route has phases, takeover points, and a replan budget. |
| **Blueprint** | The SPEC generation pipeline for a single mission. Covers all stages from input → clarification → route selection → SPEC tree → documents → preview → handoff. The `shared/blueprint/` module holds the full contract (78KB). |
| **SPEC Tree** | The central artifact of a Blueprint run. A hierarchical tree of modules where each node has: requirements, design decisions, tasks, evidence (with real provenance), and a stable ID. Validated by schema + invariant guard before being accepted. |
| **Drive State** | The high-level state machine explaining what the system is doing right now: e.g., `CLARIFYING`, `GENERATING_ROUTE`, `GENERATING_SPEC_TREE`, `WAITING_INPUT`, `REVIEWING`, `COMPLETED`. |
| **Fleet / FSD Agent Fleet** | The collection of AI roles active for a mission. In simple mode: a Single Agent. In complex mode: Planning + Architecture + Execution + Audit + UI + Critic + Grounding + Synthesizer. Dynamic org generation produces a custom fleet per task. |
| **Skill** | A hot-pluggable capability bundle for agents. Contains: system prompt, tool bindings, applicable roles, category, version. Registered in `ISkillRegistry`. Agent autonomy upgrade allows agents to self-select skills. |
| **Web-AIGC Node** | A pre-built execution node type for common AI-driven operations (OCR, translation, chart gen, vector operations, etc.). Each node has a typed input/output contract. 50+ nodes available. |
| **Lobster Executor** | The Docker-backed task execution sidecar. Receives a job envelope (image, command, credentials, resource limits), runs it in an isolated container, streams logs back, and POST callbacks with results to the server via HMAC-signed HTTP. Named "lobster" as an internal code name. |
| **Takeover Point** | An explicit human-in-the-loop pause. The system sets state to `WAITING_INPUT` and cannot proceed until the human provides input (clarification answer, route confirmation, approval, budget confirmation, or delivery review). |
| **Replan** | A mid-mission route change triggered by: new constraints, low confidence, risk signal, tool failure, quality gate failure, or user override. Replan logic is in `shared/mission/autopilot.ts`. Has a budget (max replans) and convergence threshold. |
| **Confidence** | The system's self-assessed certainty about: (a) understanding of the goal, (b) route viability, (c) execution completeness, (d) result quality. Used to decide whether to proceed or to trigger a Takeover Point or Replan. |
| **Lineage** | A DAG tracking the provenance of every data artifact: source nodes → transformation nodes → decision nodes. Operations tracked include: `llm_call`, `transform`, `enrich`, `validate`, `ml_inference`. Used for audit and impact analysis. |
| **Checks Ledger** | A JSON log (`checks_ledger.json`) where every quality gate writes its script name, exit code, and stdout/stderr. The ledger is the machine-readable proof that quality checks actually ran. |
| **Companion roles** | On-demand cross-cutting roles (Critic, Grounding) that can be injected at any stage of the pipeline. The Critic finds holes; the Grounding role forces real source citations. Both leave traces in `companion_log.json` and the checks ledger. |
| **Provenance** | The label on every generated artifact indicating how it was produced: `llm` (real LLM call), `llm_fallback` (LLM failed, used deterministic fallback), `template` (pure template). Propagated all the way to the handoff export. |
| **Reputation** | Five-dimensional score (0-1000) per agent per role: quality, speed, efficiency, collaboration, reliability. Determines trust tier (trusted/standard/probation). Used for routing decisions and external agent vetting. |
| **Guardrails / Cost Governance** | Multi-level budget system (per-mission, per-day, per-hour) with alert thresholds (0-100%), alert response strategies (LOG → REDUCE_CONCURRENCY → DOWNGRADE_MODEL → PAUSE_TASK), and model downgrade policies. |
| **NL Command** | A strategic-level instruction above missions. A single NL Command is parsed by LLM into intent + constraints → decomposed into multiple missions → execution plan with timeline, resource allocation, approval workflow, and dynamic adjustment. |
| **Scene Command** | JSON-RPC 2.0 instruction sent from the LLM director to the UE5 rendering engine over WebSocket. Methods: `character.moveTo`, `character.playAnimation`, `camera.setPreset`, `camera.transition`, `scene.setState`, `effect.play/stop`. |
| **A2A Protocol** | JSON-RPC 2.0 HTTP envelope for calling agents across different frameworks (CrewAI, LangGraph, Claude, custom). Methods: `a2a.invoke`, `a2a.stream`, `a2a.cancel`. |
| **Swarm** | Cross-pod collaboration where Manager agents from different independent pods coordinate sub-task execution. A pod can accept/reject/busy-respond to collaboration requests. |
| **Autopilot** | The L1-L5 execution level model governing how much a mission self-executes vs. asks for human input. L1 = suggestions only; L3 = standard tasks mostly auto. Level is set at mission launch and can downgrade mid-run. |

## 5. Config & Integrations (from `.env.example`)

### 5.1 Required

| Key | Purpose |
|---|---|
| `LLM_API_KEY` | OpenAI-compatible key for the primary LLM (any provider). Server boots without it but every autopilot path falls back to deterministic templates. |
| `LLM_BASE_URL` | Provider base URL (default: `https://api.openai.com/v1`). Swap for DashScope, SiliconFlow, Moonshot, Zhipu, Azure, etc. |
| `LLM_MODEL` | Model name (default: `gpt-5.4`). |
| `SESSION_SECRET` | 64-char hex secret for HMAC session cookies. |

## 7. Key Architectural Decisions Worth Noting

1. **Shared module strategy**: All contracts live in `shared/` as pure TypeScript interfaces (no runtime imports). Both client and server import from `shared/`. This eliminates the "frontend and backend drift on types" problem. For OpenReply, a shared Python dataclass / TypedDict layer between the Tauri frontend and Python sidecar would achieve the same benefit.

2. **Feature flags for every expensive/risky LLM feature**: Every LLM-backed capability has an env flag (`BLUEPRINT_SPEC_TREE_LLM_ENABLED`, etc.) defaulting to `false` or to a template fallback. This lets the app boot and demo without any keys, and lets CI always run in deterministic mode. OpenReply should adopt this — current behavior crashes if LLM keys are absent.

3. **Deterministic fallback everywhere**: WhyBuddy never returns a hard error from the SPEC pipeline. Every LLM call has a `deterministic-provider.ts` fallback that produces a structurally valid output labeled `llm_fallback`. The user gets something usable at every level.

4. **HMAC-signed executor callbacks**: The Lobster Executor signs every callback with HMAC. The server rejects unsigned or time-skewed callbacks. For OpenReply's Python sidecar callbacks (if any), this pattern prevents spoofed completion signals.

5. **Property-based testing (fast-check)**: The executor has 18+ property-based test files covering concurrency, cancellation, security policy, log batching, callback signing. This is notably thorough for an open-source AI project and explains why the executor is reliable despite being complex.

6. **Session-isolated realtime state**: All Socket.IO rooms and realtime stores are keyed by `sessionId`. Multiple browser tabs or users don't bleed state into each other. For OpenReply's multi-tab MCP + Tauri setup, session isolation is worth formalizing the same way.

7. **Lineage DAG as audit primitive**: Rather than ad-hoc logging, WhyBuddy tracks every LLM call and data transformation as a node in a lineage DAG with typed edge relationships. This makes "why did the system produce this output?" answerable programmatically. OpenReply's current logging is console-only.
